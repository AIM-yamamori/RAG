## 3. チャンク分割処理(`ingest/chunker.py`)

### 3.1 処理概要

就業規則のMarkdownファイルを読み込み、「条」単位(例:`## 第28条(年次有給休暇)`)を1チャンクとして分割する。見出しが条文単位で統一されているため、正規表現による見出し検出で分割する。

### 3.2 関数仕様

#### `parse_markdown_file(file_path: str) -> list[dict]`

| 項目 | 内容 |
|---|---|
| 引数 | `file_path`: Markdownファイルのパス |
| 戻り値 | チャンク情報の辞書のリスト |
| 処理概要 | 1. ファイルを読み込む 2. `# ` (h1) で章タイトルを抽出 3. `## 第n条(...)` の正規表現でチャンクの開始位置を検出 4. 各チャンクの本文(次の`##`見出しまで)を抽出 5. メタデータとともに辞書形式でリストに格納 |

**戻り値の構造例**

```python
[
    {
        "source_file": "04_休暇制度.md",
        "chapter_title": "第4章 休暇制度",
        "article_no": "第28条",
        "article_title": "年次有給休暇",
        "content": "## 第28条(年次有給休暇)\n\n1. 会社は、雇入れの日から6か月間継続勤務し…"
    },
    ...
]
```

### 3.3 処理ロジック詳細

```
function parse_markdown_file(file_path):
    text = ファイル全体を読み込む

    # 章タイトル抽出(h1見出し: "# 第n章 ...")
    chapter_title = 正規表現 "^# (.+)$" で最初に一致した行を抽出

    # 条文見出しの検出(h2見出し: "## 第n条(...)")
    matches = 正規表現 "^## (第[0-9]+条)\(([^)]+)\)$" で全件検出
        (グループ1: 条文番号, グループ2: 条文タイトル)

    chunks = []
    for i, match in enumerate(matches):
        start = match の開始位置
        end = 次のmatchの開始位置(最後の場合はファイル末尾)
        content = text[start:end] を取得(前後の空白をtrim)

        chunks.append({
            "source_file": ファイル名のみ抽出(パスを除く),
            "chapter_title": chapter_title,
            "article_no": match グループ1,
            "article_title": match グループ2,
            "content": content
        })

    return chunks
```

### 3.4 補足事項

- 表(Markdownテーブル)を含む条文も、条の区切り(`##`見出し)を基準に分割するため、表全体が1チャンックに収まる
- `00_概要_目次.md` は条文形式ではないため、別処理として章単位・もしくはファイル全体を1チャンクとして扱う(`parse_overview_file`関数を別途用意)
- 1チャンクが長大になる場合(目安:1,500文字超)は、Embeddingモデルのトークン上限に達する可能性があるため、念のため文字数チェックを行い、警告ログを出力する(学習用データでは想定されないが安全策として実装)

## 4. ベクトル化処理(`ingest/embedder.py`)

### 4.1 関数仕様

#### `get_embedding(text: str) -> list[float]`

| 項目 | 内容 |
|---|---|
| 引数 | `text`: ベクトル化対象のテキスト |
| 戻り値 | 768次元のfloatリスト |
| 処理概要 | Vertex AI Embedding API(text-embedding-004)を呼び出し、ベクトルを取得する |
| 例外処理 | API呼び出し失敗時は最大3回までリトライ(指数バックオフ)。3回失敗したら例外を送出し、呼び出し元(ingest.py)で該当チャンクをスキップしてログに記録する |

### 4.2 処理ロジック詳細

```
function get_embedding(text):
    for attempt in 1..3:
        try:
            response = Vertex AI Embedding APIを呼び出す(text)
            return response.embedding値(768次元のリスト)
        except APIエラー:
            if attempt == 3:
                raise EmbeddingError(f"ベクトル化に失敗しました: {text の先頭50文字}")
            wait(2 ** attempt 秒)  # 指数バックオフ
```

#### `get_embeddings_batch(texts: list[str]) -> list[list[float]]`

複数チャンクを一括処理する場合に使用。APIのバッチ呼び出し対応状況に応じて、1件ずつ`get_embedding`を呼び出すループ実装でも可とする(学習用データ件数が少ないため性能上の問題はない)。

## 5. 取り込みバッチ(`ingest/ingest.py`)

### 5.1 処理フロー

```
function main():
    1. data/fictworks_rules/ 配下の全.mdファイルを取得
    2. 各ファイルに対し chunker.parse_markdown_file() を実行し、チャンクリストを得る
    3. 全チャンクをまとめたリストを作成
    4. 各チャンクの content に対し embedder.get_embedding() を実行し、embeddingフィールドを付与
    5. db.py の insert_chunks() を呼び出し、Cloud SQLへ一括INSERT
    6. 処理結果(成功件数、失敗件数)を標準出力に表示

    if 失敗件数 > 0:
        失敗したチャンクのファイル名・条文番号を一覧表示する
```

### 5.2 実行方法

ローカル環境から、Cloud SQL Auth Proxyまたはパブリック接続(IP許可リスト方式)経由で実行する。

```bash
python -m ingest.ingest
```

### 5.3 冪等性に関する設計

再実行による重複登録を防ぐため、バッチ開始時に`rule_chunks`テーブルを`TRUNCATE`してから全件再投入する方式とする(学習用データであり差分更新の要件がないため、シンプルな全件洗い替えで十分)。

## 6. DB接続処理(`app/db.py`)

### 6.1 関数仕様

#### `get_connection()`

| 項目 | 内容 |
|---|---|
| 処理概要 | Secret Managerから接続情報を取得し、psycopg2(またはpsycopg)でCloud SQLへの接続オブジェクトを返す |
| 接続方式 | Cloud Run環境ではCloud SQL Python Connector(Unixソケット経由)を使用し、IP管理を不要にする |

#### `insert_chunks(chunks: list[dict])`

| 項目 | 内容 |
|---|---|
| 引数 | embeddingを含むチャンク辞書のリスト |
| 処理概要 | `TRUNCATE rule_chunks;` を実行後、`INSERT INTO rule_chunks (...) VALUES (...)` を全件実行してコミットする |

#### `search_similar_chunks(query_embedding: list[float], top_k: int = 3) -> list[dict]`

| 項目 | 内容 |
|---|---|
| 引数 | `query_embedding`: 質問文のベクトル、`top_k`: 取得件数(デフォルト3件) |
| 戻り値 | 類似度上位のチャンク情報のリスト(content, source_file, article_no, article_titleを含む) |
| 処理概要 | pgvectorのコサイン類似度演算子(`<=>`)を用いたSQLを実行する |

**実行SQLの例**

```sql
SELECT source_file, chapter_title, article_no, article_title, content,
       1 - (embedding <=> %(query_vector)s) AS similarity
FROM rule_chunks
ORDER BY embedding <=> %(query_vector)s
LIMIT %(top_k)s;
```

### 6.2 処理ロジック詳細

```
function search_similar_chunks(query_embedding, top_k=3):
    connection = get_connection()
    cursor = connection.cursor()
    cursor.execute(上記SQL, {"query_vector": query_embedding, "top_k": top_k})
    rows = cursor.fetchall()
    connection.close()

    return [
        {
            "source_file": row.source_file,
            "chapter_title": row.chapter_title,
            "article_no": row.article_no,
            "article_title": row.article_title,
            "content": row.content,
            "similarity": row.similarity
        }
        for row in rows
    ]
```

## 7. LLM呼び出し処理(`app/llm_client.py`)

### 7.1 関数仕様

#### `generate_answer(question: str, contexts: list[dict]) -> str`

| 項目 | 内容 |
|---|---|
| 引数 | `question`: ユーザーの質問、`contexts`: 検索された関連チャンクのリスト |
| 戻り値 | Gemini APIが生成した回答文字列 |
| 処理概要 | プロンプトテンプレートにコンテキストと質問を埋め込み、Gemini APIを呼び出して回答テキストを取得する |

### 7.2 プロンプトテンプレート

```
あなたは株式会社フィクトワークスの就業規則について回答する社内アシスタントです。
以下の「参考情報」のみを根拠として、質問に対して日本語で簡潔かつ正確に回答してください。
参考情報に記載がない内容については、推測で答えず「就業規則内に該当する情報が見つかりませんでした」と回答してください。

【参考情報】
{context_block}

【質問】
{question}

【回答】
```

`{context_block}` には、検索された各チャンクを以下の形式で連結したものを挿入する。

```
[出典: {source_file} {article_no}({article_title})]
{content}

[出典: {source_file} {article_no}({article_title})]
{content}
...
```

### 7.3 処理ロジック詳細

```
function generate_answer(question, contexts):
    context_block = ""
    for c in contexts:
        context_block += f"[出典: {c.source_file} {c.article_no}({c.article_title})]\n{c.content}\n\n"

    prompt = プロンプトテンプレート.format(context_block=context_block, question=question)

    try:
        response = Gemini API を呼び出す(prompt)
        return response.text
    except APIエラー as e:
        log.error(f"Gemini API呼び出し失敗: {e}")
        raise LLMError("回答の生成中にエラーが発生しました。")
```

### 7.4 異常系の考慮

| ケース | 対応 |
|---|---|
| Gemini APIのレート制限エラー | 1回のみ3秒待機してリトライ。再失敗時は呼び出し元にエラーを返す |
| 検索結果(contexts)が0件 | LLM呼び出しを行わず、固定文言「就業規則内に関連する情報が見つかりませんでした」を即座に返す(任意機能O-01に対応) |

## 8. RAGコアロジック(`app/rag_service.py`)

### 8.1 関数仕様

#### `answer_question(question: str) -> dict`

| 項目 | 内容 |
|---|---|
| 引数 | `question`: ユーザーの質問文字列 |
| 戻り値 | `{"answer": str, "sources": list[dict]}` |
| 処理概要 | 質問のベクトル化→類似検索→回答生成までの一連処理を統括する |

### 8.2 処理ロジック詳細

```
function answer_question(question):
    1. 入力チェック:question が空文字、または500文字超の場合は ValueError を送出

    2. query_embedding = embedder.get_embedding(question)
       # app側でもingest側と同じembedder関数を共有利用する

    3. contexts = db.search_similar_chunks(query_embedding, top_k=3)

    4. if len(contexts) == 0:
           answer = "就業規則内に関連する情報が見つかりませんでした。"
           sources = []
       else:
           answer = llm_client.generate_answer(question, contexts)
           sources = [
               {
                   "source_file": c.source_file,
                   "article_no": c.article_no,
                   "article_title": c.article_title
               }
               for c in contexts
           ]

    5. return {"answer": answer, "sources": sources}
```

### 8.3 類似度の閾値判定(任意)

検索結果は得られるが類似度が低すぎる(無関係な質問である)場合への対応として、`similarity < 0.5` のチャンクは contexts から除外するフィルタを設けることを推奨する(閾値は実装時の動作確認結果に応じて調整する)。

## 10. データモデル定義(`app/schemas.py`)

```python
from pydantic import BaseModel, Field

class ChatRequest(BaseModel):
    question: str = Field(..., min_length=1, max_length=500)

class SourceInfo(BaseModel):
    source_file: str
    article_no: str | None = None
    article_title: str | None = None

class ChatResponse(BaseModel):
    answer: str
    sources: list[SourceInfo]
```

## 11. フロントエンド処理(`app/static/index.html`)

### 11.1 主な処理(JavaScript側)

```
function sendMessage():
    questionText = 入力欄の値を取得
    if questionText が空: return

    会話表示エリアに「ユーザー: {questionText}」を追加
    入力欄をクリア
    ローディング表示を表示

    fetch("/api/chat", { method: "POST", body: JSON.stringify({question: questionText}) })
        .then(response => response.json())
        .then(data => {
            ローディング表示を非表示
            if data.error:
                会話表示エリアに「エラー: {data.error}」を追加
            else:
                answerText = data.answer
                sourcesText = data.sources を "ファイル名 条文番号" 形式で連結
                会話表示エリアに「ボット: {answerText}\n出典: {sourcesText}」を追加
        })
        .catch(error => {
            ローディング表示を非表示
            会話表示エリアに「通信エラーが発生しました」を追加
        })
```
