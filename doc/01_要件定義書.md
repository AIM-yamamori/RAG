|---|---|

| 想定期間 | 数日間(学習目的のPoC) |

## 1. プロジェクト概要

### 1.1 背景

社内ドキュメント(就業規則・ルールブック)に対して自然言語で質問すると、関連する条文を検索し、根拠を示しながら回答するRAG(Retrieval-Augmented Generation)システムを、Google Cloud上で構築する。本プロジェクトは実務利用ではなく、Google CloudおよびRAGアーキテクチャの**学習を目的としたPoC(Proof of Concept)**である。

### 1.2 目的

- Google Cloud上でのRAGシステムの一般的な構成要素(データ格納、ベクトル検索、LLM連携、Webアプリ提供)を一通り実装し、理解を深める
- Pythonを用いたクラウドネイティブなアプリケーション開発の経験を積む
- 無料クレジットの範囲内でコストを管理しながらクラウドリソースを運用する経験を積む

### 1.3 対象範囲外(スコープ外)

- 本番運用を前提とした可用性・冗長性の確保
- 大量データ・高トラフィックを想定したスケーラビリティ設計
- 認証基盤との連携(SSO等)による高度なアクセス制御
- 多言語対応(日本語のみを対象とする)
- CI/CDパイプラインの整備(任意で実施するが必須要件ではない)

## 2. 利用データ

| 項目 | 内容 |
|---|---|
| データ内容 | 架空企業「株式会社フィクトワークス」の就業規則・ルールブック(Markdown形式、全11ファイル) |
| データ件数 | 約11ファイル、条文ベースで約80条 |
| データ更新頻度 | 更新なし(静的データとして扱う) |
| 機密性 | 架空データのため機密情報は含まない |

## 3. 機能要件

### 3.1 必須機能

| ID | 機能名 | 内容 |
|---|---|---|
| F-01 | ドキュメント取り込み | Markdownファイルを読み込み、条文単位等の適切な粒度でチャンク分割する |
| F-02 | ベクトル化(Embedding) | チャンクごとにベクトル埋め込みを生成し、データベースに格納する |
| F-03 | 類似検索 | ユーザーの質問文をベクトル化し、関連性の高いチャンクをデータベースから検索する |
| F-04 | 回答生成 | 検索結果(コンテキスト)を踏まえ、LLM(Gemini API)が自然文で回答を生成する |
| F-05 | 出典表示 | 回答の根拠となった条文・ファイル名を回答とあわせて表示する |
| F-06 | チャットUI | Webブラウザ上のチャット画面で質問を入力し、回答を表示する |
| F-07 | 会話履歴表示 | 同一セッション内でのやり取りを画面上に表示する(画面をリロードすると消えてよい) |

### 3.2 任意機能(余裕があれば実装)

| ID | 機能名 | 内容 |
|---|---|---|
| O-01 | 回答未検出時のフォールバック | 関連度の高いチャンクが見つからない場合「就業規則内に該当情報が見つかりません」と回答する |
| O-02 | 取り込みバッチの自動化 | Markdown更新時に再取り込みを行うスクリプトの整備 |
| O-03 | ログ出力 | 質問・回答・検索結果をCloud Loggingに出力する |

## 4. 非機能要件

| 区分 | 要件 |
|---|---|
| コスト | Google Cloudの無料クレジット(初回特典)の範囲内に収まること。常時起動リソースは最小構成とし、使用しない期間はリソースを停止・削除できること |
| パフォーマンス | 質問入力から回答表示まで、おおむね10秒以内を目安とする(学習目的のため厳密な性能要件は設けない) |
| セキュリティ | APIキー等の秘匿情報はソースコードに直接記載せず、環境変数またはSecret Managerで管理する |
| 可用性 | 学習期間中にアクセスできれば良く、高可用性は求めない |
| 保守性 | 構成要素(取り込み・検索・生成・UI)を疎結合に保ち、後から構成変更しやすくする |
| 開発言語 | Python を主言語とする |

## 5. 利用するGoogle Cloudサービス(想定)

| サービス | 用途 | コスト方針 |
|---|---|---|
| Cloud SQL for PostgreSQL(pgvector拡張) | チャンクとベクトルデータの格納・類似検索 | 最小インスタンス(db-f1-micro相当)を使用し、不使用時は停止する |
| Vertex AI(Embedding API) | テキストのベクトル埋め込み生成 | 呼び出し回数を最小限に抑える(再生成を避ける) |
| Gemini API | 回答生成 | 無料枠の範囲内で利用 |
| Cloud Run | チャットUI(Webアプリ)のホスティング | リクエストがない時間は自動スケールダウン(課金ゼロ)させる |
| Secret Manager | APIキー・DB接続情報の管理 | 利用量が小さいため無料枠内に収まる想定 |
| Cloud Build または ローカルビルド | Cloud Runへのデプロイ | 必要最小限の利用に留める |
| Cloud Logging | 簡易なログ確認(任意機能 O-03) | デフォルトの無料枠内で利用 |

## 6. 制約条件

- 利用できる予算は、Google Cloudの無料クレジットの範囲内とする。クレジットを使い切った場合、または期間内に検証が完了しない場合はプロジェクトを終了する
- 開発・検証期間は数日間とする
- 開発者は1名を想定する
- 個人のGoogle Cloudアカウント(無料クレジット付与済み)を使用する

## 7. 受け入れ基準(完了の定義)

- 就業規則の内容について、チャット画面から日本語で質問すると、関連する条文に基づいた回答が返ってくること
- 例:「年次有給休暇は何日もらえますか?」と質問すると、第28条の内容に基づいた回答が返ること
- 回答に検索元のファイル名または条文番号が表示されること
- Cloud Run上のURLにブラウザからアクセスして動作確認ができること
- 開発期間中の利用料金が無料クレジットの範囲に収まっていること(Cloud Billingで確認)

## 8. リスクと対応方針

| リスク | 対応方針 |
|---|---|
| クラウドリソースの停止忘れによる想定外課金 | Cloud Billingの予算アラートを設定し、一定額に達したら通知を受け取る。検証終了後は速やかにリソースを削除する |
| Embedding/Gemini APIの呼び出し過多による無料枠超過 | チャンクのベクトル化は初回取り込み時のみ行い、結果を永続化して再利用する。動作確認時の質問回数を必要最小限にする |
| 学習期間内に完成しない | スコープを必須機能(3.1)に絞り、任意機能は時間に余裕がある場合のみ着手する |
